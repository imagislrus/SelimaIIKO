# Техническое задание +++111
## Агент автоматического ввода накладных в iiko

---

### 1. Общее описание

Разработка программного агента, который автоматизирует процесс внесения приходных накладных в систему iiko (веб-интерфейс). Агент сканирует/распознаёт бумажные и электронные накладные, сопоставляет позиции с номенклатурой iiko и автоматически заполняет формы в браузере.

---

### 2. Исходные данные

| Параметр | Значение |
|---|---|
| Система учёта | iiko (веб-интерфейс, браузер) |
| Формат накладных | Бумажные (фото/скан), PDF, смешанные форматы |
| Объём | 50+ накладных в день |
| Пользователи | 2–3 оператора |
| ОС операторов | Windows |
| Инфраструктура | Сеть заведений, несколько складов |
| Номенклатура | В основном заведена; иногда появляются новые позиции |

---

### 3. Функциональные требования

#### 3.1. Модуль распознавания накладных (OCR + структурирование)

**Вход:** фото, скан или PDF накладной.

**Выход:** структурированные данные в формате JSON:

```
- Номер накладной
- Дата
- Поставщик (наименование, ИНН)
- Склад / точка назначения
- Список позиций:
  - Наименование товара (как на накладной)
  - Количество
  - Единица измерения
  - Цена без НДС
  - Ставка НДС
  - Сумма
- Итого по документу
```

**Требования:**
- Локальная обработка (без облачных API) для экономии на токенах
- Поддержка русского языка
- Работа с фото разного качества (телефон, сканер)
- Работа на Windows (Apple Silicon — опционально для разработки)

**Рекомендуемый стек:**
- OCR: Tesseract OCR или PaddleOCR (локальные, бесплатные)
- Структурирование: локальная LLM через Ollama (Llama 3.2 / Mistral / Qwen) — разбирает распознанный текст в JSON
- Альтернатива: EasyOCR (Python, поддержка GPU)

#### 3.2. Модуль сопоставления номенклатуры

**Задача:** сопоставить названия товаров из накладной с позициями номенклатуры в базе iiko.

**Логика работы:**

1. При первом запуске — выгрузить справочник номенклатуры из iiko (через интерфейс или экспорт).
2. Для каждой позиции накладной — найти наиболее похожую позицию в справочнике (нечёткий поиск).
3. Показать оператору таблицу сопоставления для подтверждения:

```
| № | На накладной          | Предложено из iiko    | Совпадение | Статус     |
|---|-----------------------|-----------------------|------------|------------|
| 1 | Филе кур. охл. 1кг   | Курица филе           | 92%        | ✅ Принято  |
| 2 | Масло сл. 82.5% Простоквашино | Масло сливочное | 78%  | ⚠️ Проверить|
| 3 | Соус Терияки 0.5л     | — не найдено —        | —          | ❌ Новая    |
```

4. Запоминать подтверждённые пары в локальной базе (SQLite).
5. При повторных накладных от того же поставщика — автоматически подставлять ранее подтверждённые пары.

**Требования:**
- Нечёткий поиск (fuzzy matching) по наименованиям
- Локальная база сопоставлений с историей
- Возможность добавления новой позиции в iiko (по решению оператора)

#### 3.3. Модуль ввода в iiko (браузерный агент)

**Задача:** автоматически заполнить форму приходной накладной в веб-интерфейсе iiko.

**Последовательность действий агента:**

1. Открыть iiko в браузере (или использовать уже открытую сессию).
2. Перейти в раздел «Склад» → «Приходные накладные».
3. Нажать «Создать новую накладную».
4. Заполнить шапку:
   - Выбрать поставщика (поиск по названию/ИНН)
   - Выбрать склад / точку
   - Установить дату и номер документа
5. Добавить позиции:
   - Для каждой строки: выбрать номенклатуру, ввести количество, цену, НДС
6. Проверить итоговую сумму.
7. **Остановиться и ждать подтверждения оператора** (не сохранять автоматически).
8. После подтверждения — сохранить / провести документ.

**Рекомендуемый стек:**
- Playwright (Python) — headless или headed браузер
- Альтернатива: Selenium + ChromeDriver

**Требования:**
- Работа в фоновом режиме (не блокировать основной браузер оператора)
- Обработка ошибок (элемент не найден, таймаут, дублирование)
- Логирование всех действий
- Возможность паузы и ручного вмешательства

#### 3.4. Интерфейс оператора

**Задача:** простой UI для управления процессом.

**Основные экраны:**

1. **Загрузка накладной** — drag & drop фото/PDF или выбор файла
2. **Проверка распознавания** — отображение распознанных данных рядом с оригиналом
3. **Сопоставление номенклатуры** — таблица с предложениями и кнопками подтверждения
4. **Статус ввода** — прогресс заполнения в iiko, лог действий
5. **Очередь** — список накладных в обработке

**Рекомендуемый стек:**
- Веб-интерфейс: Streamlit (быстрый прототип) или FastAPI + React
- Альтернатива: Electron / Tauri для десктопного приложения

---

### 4. Нефункциональные требования

| Требование | Описание |
|---|---|
| Локальность | OCR и LLM работают локально, без внешних API |
| Производительность | Обработка одной накладной — не более 2–3 минут (включая ввод) |
| Отказоустойчивость | При ошибке ввода — откат и уведомление оператора |
| Многопользовательность | 2–3 оператора одновременно, каждый со своей сессией iiko |
| Масштабируемость | Поддержка нескольких точек/складов сети |
| Безопасность | Учётные данные iiko хранятся локально в зашифрованном виде |
| Логирование | Полный лог: какая накладная, когда, кем обработана, результат |

### 6. Этапы разработки

| Этап | Описание | Ориентир. срок |
|---|---|---|
| **1. Прототип OCR** | Настройка Tesseract/PaddleOCR + Ollama. Тестирование на 10–20 реальных накладных. Оценка качества распознавания. | 1–2 недели |
| **2. Сопоставление** | Выгрузка номенклатуры iiko. Реализация fuzzy matching. Создание базы пар. UI подтверждения. | 1–2 недели |
| **3. Браузерный агент** | Изучение DOM-структуры форм iiko. Скрипт Playwright для создания накладной. Обработка ошибок. | 2–3 недели |
| **4. Интерфейс** | Веб-интерфейс оператора. Очередь накладных. Статусы и логи. | 1–2 недели |
| **5. Интеграция и тест** | Сборка всех модулей. Тестирование на реальном потоке. Доработка. | 1–2 недели |
| **6. Развёртывание** | Установка на рабочие машины. Обучение операторов. Поддержка. | 1 неделя |

**Итого: 7–12 недель**

---

### 7. Риски и ограничения

| Риск | Вероятность | Решение |
|---|---|---|
| iiko обновит интерфейс — сломаются селекторы | Средняя | Модульная архитектура, быстрое обновление селекторов |
| Низкое качество фото накладных | Высокая | Предобработка изображений (контраст, поворот, обрезка) |
| Новые поставщики с нестандартным форматом | Средняя | Гибкий промпт для LLM, ручная корректировка |
| iiko заблокирует автоматизацию | Низкая | Имитация человеческого поведения, разумные паузы |
| Одновременный доступ нескольких агентов | Средняя | Очередь задач, распределение по сессиям |

### 8. Требования к инфраструктуре

**Минимальные требования к рабочей станции:**
- Windows 10/11
- 16 ГБ RAM (для локальной LLM)
- GPU (опционально, ускоряет OCR и LLM) — NVIDIA с 6+ ГБ VRAM или использование CPU-режима
- Python 3.10+
- Браузер Chrome/Chromium

**Серверный вариант (при необходимости):**
- Отдельный сервер для OCR + LLM
- Операторы работают через веб-интерфейс
- Подходит при слабых рабочих станциях
